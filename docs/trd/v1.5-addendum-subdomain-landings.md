# TRD v1.5 Addendum — Subdomain Landings (`partai` + `mufakat`)

**Date:** 2026-06-13 | **Status:** Draft | **Amends:** TRD v1.3 §11 (+ v1.4 www addendum)
**Implements:** `docs/prd/subdomain-landings-v0.1.md`

---

## 1. Purpose & relationship to existing decisions

This addendum adds a **signed-out landing** to `partai.alternatif.space` and
`mufakat.alternatif.space`, and a partai **`/cara-kerja`** deep-dive. It amends the TRD v1.3 §11
route maps in place; changes fold into the next full TRD revision.

It reconciles two prior decisions that the new landings appear to conflict with:

- **Apex-only explainer principle** (www-landing PRD §17: apex "is the only surface that explains the
  platform to someone who has not joined"). **Amended scope:** the apex remains the only *platform-level*
  explainer (the four-ruang pitch, the KBBI reframe, the cross-subdomain story). The subdomain landings
  are *product-level* explainers — they teach the mechanics of one app (how a recall works, what the
  good-question split is) that the apex deliberately does not. No duplication of the apex pitch; the
  subdomain landings link back to apex for the platform story.
- **Public-by-default content model** (TRD v1.3 §11: partai `/` = public party browse; mufakat `/` =
  public thread list; party/thread/profile pages public). **Preserved.** Content URLs stay public and
  crawlable. The landing is added as the signed-out state of `/` only; the public *content index* moves
  to a dedicated browse route so SEO and link-unfurling are unaffected (§3).

**Key change:** partai `/` is no longer "browse parties". Signed-out it is the landing; signed-in it

**Key change:** mufakat `/` signed-out = landing; signed-in = feed (unchanged behavior for logged-in
users). The *public* chronological feed is reachable at `/jelajah` for signed-out humans who want to
look around (landing "Lihat diskusi" CTA → `/jelajah`). Thread pages are unchanged and remain the
primary SEO surface.

## 3. The X-style public-content model (SEO without an auth wall)

**Goal:** logged-out visitors and crawlers can read content (like X serves public posts), while `/`
shows a landing to humans and the personalized feed only renders for signed-in users.

How X does it, and how we mirror it:

| Concern | X behavior | Our implementation |
|---|---|---|
| Crawlable content | Post/profile URLs are SSR'd HTML with OG/Twitter-card meta + sitemaps | `/diskusi/[slug]`, `/partai/[slug]`, `/profil/[username]`, `/jelajah` are **public + SSR/prerendered**, each with per-page OG meta (already specced: partai P0-04 OG tags) |
| Home is personalized | `/home` requires auth | mufakat `/` signed-in feed reads from auth context; signed-out it's the static landing — no personalized data fetched |
| Soft gate, not hard wall | Logged-out readers see content + sign-up nudges | Public content routes render fully for signed-out users; a sign-up nudge component may overlay (non-blocking) — **no redirect-to-login on content routes** |
| Discoverability | Sitemap + OG | Generate `sitemap.xml` listing public party + thread + open-question URLs; keep `<link rel=canonical>` per page (apex pattern) |

**Implementation notes:**

- **Rendering:** content routes use SvelteKit SSR (or prerender where data is static enough); the
  Cloudflare Pages deployment already serves SSR. The landing routes (`/` signed-out, `/cara-kerja`)
  are **prerendered** — no auth call, no DB read, fast and crawlable.
- **Auth branch at `/`:** the root `+page` resolves Clerk session server-side. Signed-out → render
  landing (prerendered/static). Signed-in → render dashboard (partai) / feed (mufakat). This is a
  render branch, **not** an HTTP redirect, so crawlers hitting `/` get the landing HTML (indexable
  marketing copy) and authed users get their app.
- **No auth wall on content:** `/jelajah`, `/diskusi/[slug]`, `/partai/[slug]`, `/profil/[username]`
  must never `redirect(...)` unauthenticated users to login. Write actions (reply, create, react,
  sign petition) are gated; reading is not.
- **robots:** landing + content routes `index,follow`. Keep `/masuk`, `/daftar`, `/moderasi*`, and
  authenticated `/dashboard*` as `noindex` (consistent with the www addendum's treatment of
  `/daftar`).

## 4. Auth, gating & cross-subdomain SSO (no change to the model)

- Clerk JWT scoped to `.alternatif.space` already gives cross-subdomain SSO (TRD §… auth). A user
  signed in on apex or partai is signed-in on mufakat — so the landing only ever shows to genuinely
  signed-out visitors.
- **partai `/` signed-in but no party joined:** the dashboard needs a selected party. If the user has
  no membership, `/` redirects to `/bergabung` (the existing onboarding gate) rather than rendering an
  empty dashboard. (Mirrors TRD v1.3 §11 `/bergabung` semantics.)
- **partai `/` signed-in, multiple parties:** dashboard renders the *selected* party; party switching
  is an existing dashboard concern, out of scope for this addendum.
- **Onboarding still owned by partai** (P0-01). The subdomain landings' sign-up CTAs use Clerk
  `<SignUp />` with `forceRedirectUrl` → `https://partai.alternatif.space/onboarding`, identical to
  the apex pattern (www addendum §). The landings never duplicate onboarding.

## 5. Design system reuse

Both landings reuse the apex `apps/www` Pamflet × Arena system rather than re-deriving it:

- Tokens: `--lp-bone`, `--lp-ink`, `--lp-cap` (stamps only), `--lp-amber*`. Fonts: `@fontsource/archivo-black`,
  `@fontsource-variable/space-grotesk`. Patterns: `lp-display`, `lp-stamp`, `lp-cell`, KBBI inverted
  band, tally divider, `use:reveal` motion gated on scripting + reduced-motion.
- **Recommendation:** promote the apex `lp-*` styles + `reveal.ts` into `packages/ui` so all three
  surfaces share one source of truth instead of copy-pasting CSS. (Currently the styles live inline in
  `apps/www/src/routes/+page.svelte`.) Tracked as a follow-up, not a blocker.
- mufakat's "Ini bukan media sosial" band reuses the inverted-ink KBBI treatment; partai's
  `/cara-kerja` governance table reuses the `aturan` definition-list pattern.

## 7. Open technical questions

- **TQ1 — `/jelajah` vs. keeping browse at `/`.** This addendum moves public browse off `/` so `/`
  can be landing/dashboard. Alternative: keep browse at `/` for signed-out and put the landing at
  `/tentang`. Chosen approach matches the user's "signed-out sees landing" intent and the apex
  pattern; flag if the existing `apps/*/src/routes/+page.svelte` (already built as browse/feed) makes
  the `/tentang` split cheaper to ship.
- **TQ2 — prerender vs SSR for signed-out `/`.** Prerender is fastest + most crawlable but means the
  signed-out/signed-in branch happens client-side after hydration (flash risk). Alternative: SSR `/`
  with the Clerk session read server-side (no flash, slight latency). Recommend SSR `/` with a cached
  landing shell. Decide during build.
- **TQ3 — sign-up nudge overlay on content routes.** Whether signed-out readers on `/diskusi/[slug]`
  get a non-blocking sign-up prompt (X-style). Out of scope for v0.1; note as a Phase 1 conversion lever
  (ties to www-landing W-Q3 analytics deferral).
- **TQ4 — sitemap ownership.** One sitemap per subdomain vs. a sitemap index at apex. Recommend
  per-subdomain `sitemap.xml`, referenced from each `robots.txt`.
