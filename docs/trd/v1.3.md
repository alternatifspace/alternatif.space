## 1. Overview

alternatif.space is a civic technology platform structured across four subdomains: partai (party formation), mufakat (discussion), simposium (live deliberation), and perpus (documentation). Party membership on partai acts as a persistent ideological flag carried across all subdomains.

This TRD covers the technical implementation across all four subdomains. partai and mufakat launch together as a single Phase 0 unit (joint launch dependency — partai PRD v0.3, mufakat PRD v0.1). The other subdomains follow once the previous phase is validated.

The TRD implements **partai PRD v0.3** and **mufakat PRD v0.1** across Phase 0, Phase 1, and Phase 2. Only Phase 0 is built first. Phases 1 and 2 are documented here for architectural foresight — no Phase 1 or 2 tables, functions, or components are built until the previous phase is validated.

## 2. Tech Stack

| Layer | Choice | Notes |
|---|---|---|
| Framework | SvelteKit | All four subdomains. One framework, one mental model. |
| Styling | Tailwind CSS | Utility-first, mobile-first |
| Auth | Clerk | JWT scoped to `.alternatif.space` for cross-subdomain SSO |
| Database | Supabase (PostgreSQL) | RLS enforced at database level |
| Postgres Extensions | `pg_trgm` (Phase 0), `pgvector` (Phase 2) | Trigram similarity powers mufakat dedup with zero AI. pgvector is not enabled until Phase 2. |
| Full-Text Search | Postgres FTS, `indonesian` config | Snowball Indonesian stemmer (PG 13+); used alongside pg_trgm for dedup suggestions |
| Storage | Supabase Storage | Party logos, share cards, future assets |
| Rich Text Editor | TipTap (Svelte) | Manifesto creation/editing; mufakat thread and comment composition |
| Diff Library | diff-match-patch | Manifesto amendment diffs (Phase 1) |
| Web Deployment | Cloudflare Pages | Best free tier; SE Asia edge nodes. One deployment per subdomain. |
| Native Wrapper | Capacitor | Single app wrapping all four subdomains for app store distribution |
| Android Distribution | Play Store via Capacitor | One listing. Trust signal for Indonesian users. |
| iOS Distribution | App Store via Capacitor | One listing. Apple review required. |
| Push Notifications | Capacitor (FCM + APNs) | Web Push as fallback for browser users |
| Shared UI | `packages/ui` (Svelte components) | Shared across all four SvelteKit apps |
| Automation | GitHub Actions | Scheduled cron jobs for vote closing, dormant status, OP-window expiry |

**Why SvelteKit for all subdomains (previously Astro for mufakat and perpus):**
The original multi-framework decision predated the native app distribution requirement. Four separate Capacitor apps — one per subdomain — would require four separate app store listings and force users to download multiple apps, which kills adoption. A single Capacitor app requires a single framework. SvelteKit handles SSR, static output (`adapter-static` for perpus), and lazy hydration well enough that Astro's island architecture advantage does not justify the added complexity. One framework, one Capacitor app, one app store listing per platform.

## 4. Distribution & Deployment

### Web
Each subdomain is a separate SvelteKit app deployed independently to Cloudflare Pages. They share auth (Clerk cookie) and database (Supabase) but are deployed as distinct units.

```
partai:
  Build command: pnpm build
  Output directory: .svelte-kit/cloudflare
  Adapter: @sveltejs/adapter-cloudflare
  Custom domain: partai.alternatif.space

mufakat:
  Build command: pnpm build
  Output directory: .svelte-kit/cloudflare
  Adapter: @sveltejs/adapter-cloudflare
  Custom domain: mufakat.alternatif.space

simposium:
  Build command: pnpm build
  Output directory: .svelte-kit/cloudflare
  Adapter: @sveltejs/adapter-cloudflare
  Custom domain: simposium.alternatif.space

perpus:
  Build command: pnpm build
  Output directory: build/
  Adapter: @sveltejs/adapter-static
  Custom domain: perpus.alternatif.space
```

### Native (Capacitor)
A single `native/` Capacitor app wraps all four subdomains under one binary. Navigation between subdomains happens within the app — no external browser required. One Play Store listing, one App Store listing.

```
Each SvelteKit app builds independently to static files
  ↓
native/capacitor.config.ts points to the live web URLs (or bundled builds)
  ↓
npx cap sync copies assets into Android/iOS native projects
  ↓
Android Studio → Play Store
Xcode → App Store
```

The Capacitor shell uses in-app WebViews per subdomain with a native bottom tab bar or drawer navigation for switching between partai, mufakat, simposium, and perpus. This gives users a unified app experience while keeping subdomain codebases independent.

App Store submission frames the platform as a civic education app. Review timeline should be factored into launch planning — Apple review typically takes 1–3 days but can be longer for first submissions.

### Environment Variables
Stored in Cloudflare Pages dashboard per subdomain for web, and in native build config for Capacitor:
```

PUBLIC_SUPABASE_ANON_KEY
PUBLIC_CLERK_PUBLISHABLE_KEY
CLERK_SECRET_KEY
```

## 5. Auth & Cross-Subdomain Session

Clerk handles authentication. The JWT is scoped to `.alternatif.space` (root domain cookie) so a user authenticated on `partai.alternatif.space` is automatically authenticated on all other subdomains without re-login.

```
User logs in via Clerk
  ↓
Clerk issues JWT
  ↓
Cookie set: domain=.alternatif.space
  ↓
mufakat, simposium, perpus all read the same cookie
  ↓
Each subdomain initialises Supabase client with Clerk JWT
  ↓
Supabase RLS uses auth.uid() from the JWT
```

Clerk JWT template must be configured to produce a Supabase-compatible token with `sub` mapped to the user's Supabase UUID.

**Party membership flag in other subdomains:**
Any subdomain determines a user's current party with a single query:
```sql
SELECT party_id FROM party_members
WHERE user_id = auth.uid() AND status = 'active'
LIMIT 1;
```
No special sync or replication needed.

**Flag-at-time-of-posting (mufakat):** mufakat never joins live to `party_members` for display. The author's party at posting time is denormalised onto each thread/comment row (`*_party_id`). Leaving a party later does not strip historical flags — consistent with partai P0-11 and mufakat M0-06. Muted members (`party_members.status = 'muted'`) post with `author_party_id = NULL`; mufakat renders the flagless state from day one.

### mufakat — Phase 0 Tables (build now) *(new v1.3)*

All mufakat tables are prefixed `mufakat_` — the database is shared across subdomains and simposium will likely also have threads/comments.

#### `mufakat_threads`
```sql
CREATE TABLE mufakat_threads (
  id               uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  slug             text UNIQUE NOT NULL,
  title            text NOT NULL CHECK (char_length(title) <= 200),
  body_content     jsonb,                   -- TipTap JSON
  body_html        text,                    -- rendered HTML
  body_text        text,                    -- plain text extraction, feeds search_tsv
  op_id            uuid REFERENCES users(id),          -- NULL for community-raised threads
  op_party_id      uuid REFERENCES parties(id),        -- flag at time of posting
  community_raised boolean DEFAULT false,   -- OP-ship declined (M0-09)
  status           text DEFAULT 'aktif'
                     CHECK (status IN ('aktif', 'selesai', 'pertanyaan_terbuka', 'dialihkan')),
  origin           text DEFAULT 'user'
                     CHECK (origin IN ('user', 'split', 'spinoff')),
  parent_thread_id uuid REFERENCES mufakat_threads(id),  -- set for split/spinoff origins
  seed_comment_id  uuid,                    -- FK added after mufakat_comments exists
  redirect_to      uuid REFERENCES mufakat_threads(id),  -- set when status = 'dialihkan'
  closing_summary  jsonb,                   -- TipTap JSON, set when status = 'selesai'
  search_tsv       tsvector GENERATED ALWAYS AS (
                     to_tsvector('indonesian',
                       coalesce(title, '') || ' ' || coalesce(body_text, ''))
                   ) STORED,
  created_at       timestamptz DEFAULT now(),
  closed_at        timestamptz
);

CREATE INDEX mufakat_threads_tsv_idx  ON mufakat_threads USING gin (search_tsv);
CREATE INDEX mufakat_threads_trgm_idx ON mufakat_threads USING gin (title gin_trgm_ops);
```

Status mapping forward to Phase 1 full lifecycle (mufakat PRD M0-03): `aktif → Pending family`, `selesai → Integrated family`, `pertanyaan_terbuka → Axiom Stop family`. The Phase 0 enum is a strict subset — migration is additive, no remapping.

#### `mufakat_comments`
```sql
CREATE TABLE mufakat_comments (
  id                 uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  thread_id          uuid REFERENCES mufakat_threads(id) NOT NULL,
  author_id          uuid REFERENCES users(id) NOT NULL,
  author_party_id    uuid REFERENCES parties(id),    -- NULL when author was muted at post time
  parent_id          uuid REFERENCES mufakat_comments(id),
  depth              int NOT NULL DEFAULT 0 CHECK (depth BETWEEN 0 AND 3),
  content            jsonb,                  -- TipTap JSON
  html               text,
  body_text          text,
  state              text DEFAULT 'visible'
                       CHECK (state IN ('visible', 'deleted', 'hidden', 'moved')),
  moved_to_thread_id uuid REFERENCES mufakat_threads(id),  -- set when state = 'moved'
  edited_at          timestamptz,
  created_at         timestamptz DEFAULT now()
);

ALTER TABLE mufakat_threads
  ADD CONSTRAINT mufakat_threads_seed_comment_fk
  FOREIGN KEY (seed_comment_id) REFERENCES mufakat_comments(id);
```

State semantics:
- `visible` — normal
- `deleted` — author soft delete; render "[dihapus]", structure preserved (replies and splits depend on positional integrity)
- `hidden` — moderator action; render "[disembunyikan moderator]", structure preserved
- `moved` — this row is the **scar** of a good-question split (mufakat PRD M0-10/M0-11). Full text is preserved on the row for audit; the render layer shows only a two-line excerpt card. The canonical full text lives in the target thread. Row is frozen — no further edits, reactions, or replies; reply attempts redirect to `moved_to_thread_id`.

`depth` is computed server-side on insert (`parent.depth + 1`, top-level = 0). The CHECK enforces max depth 3 — deeper exchanges are spin-off candidates, not infinite nesting (M0-05).

#### `mufakat_reactions`
```sql
CREATE TABLE mufakat_reactions (
  id         uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  comment_id uuid REFERENCES mufakat_comments(id),
  thread_id  uuid REFERENCES mufakat_threads(id),
  user_id    uuid REFERENCES users(id) NOT NULL,
  type       text NOT NULL CHECK (type IN ('setuju', 'pertanyaan_bagus')),
  created_at timestamptz DEFAULT now(),
  CHECK ((comment_id IS NULL) <> (thread_id IS NULL)),   -- exactly one subject
  CHECK (type = 'setuju' OR comment_id IS NOT NULL),     -- pertanyaan_bagus only on comments
  UNIQUE (comment_id, user_id, type),
  UNIQUE (thread_id, user_id, type)
);
```
No downvote type exists, by design (M0-07).

#### `mufakat_splits`
Audit record and state machine for every split, spin-off, and dedup reference.
```sql
CREATE TABLE mufakat_splits (
  id                       uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  type                     text NOT NULL
                             CHECK (type IN ('good_question', 'spinoff', 'dedup_reference')),
  source_thread_id         uuid REFERENCES mufakat_threads(id) NOT NULL,
  source_comment_id        uuid REFERENCES mufakat_comments(id),  -- seed (good_question) or first triggering comment (spinoff)
  target_thread_id         uuid REFERENCES mufakat_threads(id) NOT NULL,
  reaction_count_at_split  int,
  participants_at_split    int,
  threshold_at_split       int,
  op_status                text DEFAULT 'pending'
                             CHECK (op_status IN ('pending', 'confirmed', 'declined', 'auto_confirmed', 'n_a')),
  op_window_ends_at        timestamptz,         -- now() + 24h for good_question; NULL otherwise
  created_by               uuid REFERENCES users(id),  -- NULL = automatic; admin id for spinoff/merge
  created_at               timestamptz DEFAULT now()
);
```
`type = 'dedup_reference'`: the threshold was met but a canonical thread for the question already exists — no new thread was created; `target_thread_id` points at the existing canonical thread and the seed comment's `moved_to_thread_id` points there too. Phase 0: this path requires admin confirmation before executing (M0-09); pending candidates sit in the admin queue.

#### `mufakat_markers`
Inline markers rendered inside a thread at the point where a spin-off or redirect occurred. (Good-question placeholders are **not** markers — they are the seed comment row itself with `state = 'moved'`.)
```sql
CREATE TABLE mufakat_markers (
  id               uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  thread_id        uuid REFERENCES mufakat_threads(id) NOT NULL,  -- thread the marker renders in
  after_comment_id uuid REFERENCES mufakat_comments(id),          -- position; NULL = top of thread
  split_id         uuid REFERENCES mufakat_splits(id) NOT NULL,
  kind             text NOT NULL CHECK (kind IN ('spinoff', 'dedup_reference')),
  label            text,                    -- one-line description of the disputed question
  created_at       timestamptz DEFAULT now()
);
```
There is no `summary_back` kind and no copied summary. The summary-back block (M0-15) is rendered **live**: the marker component joins through `split_id → target_thread_id` and reads the target thread's current `status` and `closing_summary` at render time. One source of truth; a disputed summary changes in exactly one place. The same live join powers the status badge on good-question placeholder cards.

## 8. Dormant Status Logic

`status` is a stored value in the database — the source of truth for all subdomains. Two mechanisms keep it current:

**Trigger on `leader_last_active_at`** handles real-time cases — when a leader logs in, their activity timestamp updates and the trigger immediately re-evaluates dormant status:

```sql
CREATE OR REPLACE FUNCTION update_party_dormant_status()
RETURNS TRIGGER AS $$
BEGIN
  UPDATE parties
  SET status = CASE
    WHEN NEW.leader_last_active_at < now() - interval '30 days'
      AND status = 'active' THEN 'dormant'
    WHEN NEW.leader_last_active_at >= now() - interval '30 days'
      AND status = 'dormant' THEN 'active'
    ELSE status
  END
  WHERE id = NEW.id;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

**Daily GitHub Actions cron** handles the cold case — leaders who go silent and never trigger the above (workflow in Section 10).

## 9. Split & Spin-off Mechanics *(new v1.3)*

The mufakat structural mechanics are multi-table, side-effecting writes. They live in Edge Functions, never in client calls and never in database triggers — consistent with the existing convention (Section 10) and because split execution involves notification dispatch and config reads that don't belong in trigger context.

### 9.1 Threshold evaluation (M0-08)

Evaluated inside the `mufakat-react` Edge Function on every successful `pertanyaan_bagus` insert:

```sql
-- unique participants = OP + distinct comment authors in the canonical thread
WITH participants AS (
  SELECT op_id AS uid FROM mufakat_threads WHERE id = $thread_id AND op_id IS NOT NULL
  UNION
  SELECT DISTINCT author_id FROM mufakat_comments
  WHERE thread_id = $thread_id AND state <> 'moved'
)
SELECT count(*) FROM participants;
```

```
config    := platform_config['mufakat_split_threshold']   -- {floor: 5, ceiling: 50, ratio: 0.15}
threshold := GREATEST(config.floor,
             LEAST(config.ceiling, CEIL(config.ratio * unique_participants)))
```

If `reaction_count >= threshold` and no `mufakat_splits` row exists for this comment → proceed to split execution. The threshold value is never returned to clients (structure that doesn't show itself); only the reaction count is public.

### 9.2 Good-question split execution (M0-09, M0-10, M0-11)

Single transaction:

```
1. Dedup check: search_similar_threads(comment.body_text)
   ├─ candidate found (Phase 0) → INSERT mufakat_splits
   │    (type='dedup_reference', op_status='n_a', target = candidate)
   │    flagged pending admin confirmation; STOP — no new thread.
   │    On admin confirm: seed comment → state='moved',
   │    moved_to_thread_id = candidate. On reject: proceed to step 2.
   └─ no candidate → continue

2. INSERT mufakat_threads
     origin='split', parent_thread_id = source thread,
     seed_comment_id = comment.id,
     title derived from comment (admin/OP-editable later),
     body = comment content + auto-generated context block
            (quote/excerpt of parent OP, link back),
     op_id = comment author, op_party_id = comment.author_party_id

3. UPDATE mufakat_comments SET thread_id = new_thread, parent_id = NULL | rethreaded
     WHERE id IN (descendants of seed comment)
   -- replies MOVE. Nothing is duplicated (D0: "Move, leave a scar").
   -- depth recomputed relative to the new thread.

4. UPDATE seed comment: state='moved', moved_to_thread_id = new_thread
   -- row frozen; full text retained for audit; rendered as excerpt card
   -- (excerpt + link + live status badge + reaction count — D4)

5. INSERT mufakat_splits
     (type='good_question', reaction/participant/threshold snapshot,
      op_status='pending', op_window_ends_at = now() + 24h)

6. INSERT notification → comment author ("Komentarmu jadi diskusi baru")
```

**OP-ship window (D2):** the split happens regardless of the author. Within 24 hours the author confirms or declines via `mufakat-respond-opship`:
- confirm → `op_status='confirmed'`
- decline → `op_status='declined'`; thread `op_id = NULL`, `community_raised = true` ("Diangkat komunitas", admin-stewarded; seed comment attribution untouched)
- no response → `expire-op-windows` cron sets `op_status='auto_confirmed'` silently

**Recursion (D3):** there is no depth limit and no special-case code. A split thread is an ordinary thread; the same functions apply at every level. Depth is observable via the `parent_thread_id` chain and tracked as an analytics signal only.

### 9.3 Admin spin-off execution (M0-13)

Same transaction shape, admin-initiated (`is_platform_admin()` enforced):

```
1. Admin selects a contiguous reply chain constituting the semantic sub-debate
2. Dedup check against existing canonical threads
   ├─ exists → replies MOVE to the existing thread; marker links there
   └─ not    → new thread created (origin='spinoff'); replies MOVE into it
3. Triggering comments STAY in the original thread (unlike good-question seeds)
4. INSERT mufakat_markers (kind='spinoff', after_comment_id = last triggering
   comment, label = one-line description of the disputed question)
5. INSERT mufakat_splits (type='spinoff', op_status='n_a', created_by = admin)
6. INSERT mufakat_moderation_log ('spinoff_executed')
```

### 9.4 Summary-back (M0-15)

No write occurs in the parent thread when a spin-off concludes. The marker component reads the target thread's `status` and `closing_summary` live at render time — when the spin-off reaches `selesai`, the summary block appears in the parent automatically; when it reaches `pertanyaan_terbuka`, the marker shows the open-question state. Disputing a summary routes into the spin-off thread (Phase 1: `mufakat_summary_disputes`); the parent has nothing to relitigate because the parent stores nothing.

### 9.5 Dedup at creation (M0-02)

A read-only Postgres RPC, called as the user types a title (debounced client-side):

```sql
CREATE OR REPLACE FUNCTION search_similar_threads(q text)
RETURNS TABLE (id uuid, slug text, title text, status text, sim real)
LANGUAGE sql STABLE AS $$
  SELECT id, slug, title, status, similarity(title, q) AS sim
  FROM mufakat_threads
  WHERE status <> 'dialihkan'
    AND (similarity(title, q) > 0.3
         OR search_tsv @@ plainto_tsquery('indonesian', q))
  ORDER BY sim DESC
  LIMIT 5;
$$;
```

Suggestions are an invitation, never a block — the user can always proceed. No embeddings, no AI, no external calls in Phase 0; Phase 2 swaps the internals of this function for pgvector similarity without changing its signature (M2-02).

### 9.6 Rate limiting (M0-17)

Enforced inside `create-thread` / `create-comment` Edge Functions by counting the caller's rows in the relevant window against `platform_config['mufakat_rate_limits']`. No dedicated table. Limits are invisible until hit, then explained plainly in the error payload.

## 10. Edge Functions

Complex writes that touch multiple tables or require threshold checks go through Supabase Edge Functions, not direct client calls.

### partai — Phase 0 Functions

**`join-party`** — Checks user has no current active membership, inserts into `party_members`, writes to `party_membership_history`.

**`leave-party`** — Removes from `party_members`, writes leave record to `party_membership_history`.

**`generate-invite-token`** — Creates a cryptographically random token, inserts into `party_invites` with 7-day expiry.

**`use-invite-token`** — Validates token (exists, not used, not expired), triggers join flow, marks token used.

**`submit-application`** — Inserts application. Checks user has no existing pending application for the same party.

**`review-application`** — Leader approves or rejects. On approval triggers join flow. Checks caller is party leader or delegated role.

**`update-dormant-status`** — Called by daily GitHub Actions cron. Updates all parties where `leader_last_active_at` has crossed the 30-day threshold.

### mufakat — Phase 0 Functions *(new v1.3)*

**`mufakat-create-thread`** — Rate-limit check, party-membership check, slug generation, plain-text extraction from TipTap JSON, insert. (Dedup suggestions happen client-side pre-submit via the `search_similar_threads` RPC.)

**`mufakat-create-comment`** — Rate-limit check, thread `aktif` check, depth computation and max-depth enforcement, redirect rejection if the parent is a `moved` scar (error payload carries `moved_to_thread_id` so the client can route the user to the new thread).

**`mufakat-react`** — Inserts reaction. On `pertanyaan_bagus`: evaluates the threshold (9.1) and, if met, executes the split (9.2) within the same invocation.

**`mufakat-respond-opship`** — Author confirms or declines OP-ship within the 24h window (9.2).

**`expire-op-windows`** — Cron: auto-confirms `pending` splits past `op_window_ends_at`.

**`mufakat-admin-spinoff`** — Admin-only spin-off execution (9.3).

**`mufakat-confirm-dedup-reference`** — Admin confirms or rejects a pending `dedup_reference` split (9.2 step 1).

**`mufakat-close-thread`** — OP or admin sets `selesai` (with closing summary) or `pertanyaan_terbuka`. Parent markers reflect the change automatically via live read — no write-back.

**`mufakat-merge-threads`** — Admin closes a duplicate: status `dialihkan`, `redirect_to` set, optional comment move, moderation log entry.

**`mufakat-report-content`** / **`mufakat-review-report`** — Report intake; admin review (dismiss / hide / escalate). Hide sets `state='hidden'` and writes `mufakat_moderation_log`.

### Phase 1 Functions

**`cast-vote`** — Inserts vote, checks if threshold is now met, if so closes proposal and executes outcome atomically (manifesto update, leadership transfer, etc.).

**`sign-petition`** — Inserts signature, checks if threshold is met, if so auto-creates a recall vote proposal.

**`close-expired-proposals`** — Called every 15 minutes by GitHub Actions cron. Tallies results on any proposal past `voting_ends_at`, updates status, executes outcome.

**`send-push-notification`** — Called internally when a notification is created. Looks up user's `push_subscriptions`, dispatches via FCM (Android), APNs (iOS), or Web Push (browser fallback).

**`mufakat-set-argument-status`** / **`mufakat-route-open-question`** / **`mufakat-dispute-summary`** — Phase 1 resolution infrastructure (M1-01, M1-04, M1-05).

### Thread Page Rendering (mufakat) *(new v1.3)*

The thread page interleaves three row types, all positionally anchored:

1. **Comments** — from `mufakat_comments_public`, rendered by state (`visible` → full; `deleted` → "[dihapus]" stub; `hidden` → "[disembunyikan moderator]" stub).
2. **Placeholder cards** (`state = 'moved'`) — `SplitPlaceholderCard.svelte`: `moved_excerpt` (two lines), "→ kini diskusi tersendiri" link, **live** `ThreadStatusBadge` resolved by joining `moved_to_thread_id → mufakat_threads.status`, and `reaction_count_at_split` from the split row. 3. **Spin-off markers** (`mufakat_markers`, positioned via `after_comment_id`) — label + link + live status badge. The threshold formula is never shipped to or computed on the client. Sort is chronological; there is no engagement-ranked default.

TipTap is loaded only on `/buat` and on comment-composer mount — not bundled into the thread-read path (consistent with the performance budget in Section 14).

### Create Party Flow (Multi-step)

Step 1 (starting point) → Step 2 (parameters) → Step 3 (identity) → Step 4 (council) → Step 5 (review) → Publish

State held in a single local store for the duration of the flow. Nothing written to the database until the final publish step. On publish, `governance_config` is frozen and `honeymoon_ends_at` is set to `now() + interval '3 months'`.

**Step 1 — Four options:**
- Vanguard, Republic, Commune: pre-fill the local store's parameter fields with preset values, then navigate to Step 2
- Custom: navigate to Step 2 with all parameter fields in the local store left at `null`

**Step 2 — Two states depending on path:**
- Preset path: all fields populated, CTA active immediately, founder can proceed without touching anything
- Custom path: all fields `null`, CTA disabled, enabled only once every parameter field has a non-null value. A derived store counts unset fields and surfaces the completion counter in the UI.

```javascript
// Derived store for Custom path completion
const unsetCount = derived(createPartyStore, ($s) =>
  Object.values($s.governance_config).filter(v => v === null).length
);
// CTA disabled when unsetCount > 0
```

**Archetype is never written to the store or database.** It is used only to determine which pre-fill values to apply in Step 1.

### Manifesto Diff (Phase 1)

Amendment proposals render a diff using `diff-match-patch` against plain text extracted from TipTap JSON. A "lihat versi lengkap" toggle switches to side-by-side view for members who find the diff hard to read.

```javascript
import { diff_match_patch } from 'diff-match-patch';

const dmp = new diff_match_patch();
const diffs = dmp.diff_main(
  tipTapToPlainText(currentManifesto),
  tipTapToPlainText(proposedManifesto)
);
dmp.diff_cleanupSemantic(diffs);
// render with red/green highlighting
```

## 15. Content Moderation

**partai (Phase 0):** Manual approval for all new party creations. Moderator reviews party name, tagline, and manifesto before the party appears in browse. Target review time: 24 hours. Moderation queue is a Supabase dashboard view filtered on `parties.status = 'pending_review'` — no custom admin UI for partai Phase 0.

**mufakat (Phase 0) *(new v1.3)*:** Post-moderation via report-and-review (M0-16) — threads and comments publish immediately, no pre-approval queue (pre-approving live deliberation would kill it).
- Report intake: category (sara / defamation / threat / spam) + optional note; confirmation copy explains the legal basis of each category — ITE constraints as pedagogy at the point of friction, never a wall of rules at onboarding.
- Admin actions: dismiss / hide / escalate. Hide preserves structure (`state='hidden'`).
- **Minimal custom admin UI is required for mufakat** (`/moderasi` routes): spin-off execution, dedup-reference confirmation, and reply-chain selection cannot be performed from the Supabase dashboard. This supersedes the "no custom admin UI" rule for mufakat scope only; partai Phase 0 moderation stays in the dashboard.
- Public accountability: `/moderasi/log` renders the aggregate moderation log (action type + date; no reporter or moderator identity) — consistent with the platform transparency principle and perpus's eventual role.

**Compliance baseline:** ITE Law — no blasphemy, no ethnic hate speech, no violence incitement. These are the only hard-coded rules. All other content rules emerge through the platform's own constitutional process.

## 16. Resolved Architecture Decisions

| # | Question | Decision |
|---|---|---|
| Q1 | Scheduled job for closing expired votes | GitHub Actions every 15 min → Supabase Edge Function |
| Q2 | Dormant status — read-time vs stored | Stored value; Postgres trigger on activity + daily GitHub Actions cron |
| Q3 | Manifesto diff format | `diff-match-patch` on plain text; side-by-side toggle as fallback |
| Q4 | Push notifications | Capacitor (FCM/APNs) primary; Web Push fallback for browser users; in-app bell at launch (moved to Phase 0 in v1.3 — OP-ship window depends on it) with push as fast follow |
| Q5 | PWA vs native | PWA + Capacitor wrapper for Play Store and App Store distribution |
| Q6 | Multi-framework vs SvelteKit-only | SvelteKit for all four subdomains. Four separate Capacitor apps = four app store listings = unacceptable user friction. One Capacitor shell requires one framework. |
| Q7 | Archetype label visibility | Never stored, never returned by any API, never shown publicly. UI-only founding tool that pre-fills the parameter form. Only raw `governance_config` values are stored and displayed. |
| Q8 *(new)* | mufakat dedup implementation (Phase 0) | `pg_trgm` similarity + Postgres FTS (`indonesian` config) via a stable RPC. Zero AI, zero external calls. Phase 2 swaps the RPC internals for pgvector without changing its signature. |
| Q9 *(new)* | Where the split threshold is evaluated | Inside the `mufakat-react` Edge Function on `pertanyaan_bagus` insert — not a DB trigger (split execution has multi-table side effects, notifications, and config reads that don't belong in trigger context). Constants live in `platform_config`, tunable without release. Threshold math is never shipped to clients. |
| Q10 *(new)* | Move semantics (D0: "Move, leave a scar") | Replies move via `UPDATE thread_id` in one transaction — no duplication anywhere. The seed comment row is frozen as the scar (`state='moved'`): full text retained for audit, render layer shows a two-line excerpt card. Reply attempts on a scar redirect to the target thread. |
| Q11 *(new)* | Summary-back storage | Never copied. Markers and placeholder cards resolve the target thread's `status` and `closing_summary` **live at render time**. One source of truth — a disputed summary changes in exactly one place, and parent threads have nothing to relitigate. |
| Q12 *(new)* | Hidden/deleted content exposure | Rows stay selectable (positional integrity for replies, markers, splits) but content is withheld by state via the `mufakat_comments_public` view. Clients never read the base comments table. |
| Q13 *(new)* | Platform admin model | `platform_admins` table + `is_platform_admin()` SQL helper, enforced in RLS and Edge Functions. Minimal custom admin UI in the mufakat app for spin-off execution and queues — partai moderation remains Supabase-dashboard-only in Phase 0. |
