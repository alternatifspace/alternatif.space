# TRD v1.4 Addendum — `apps/www` (Root-Domain Landing)
**Date:** June 2026 | **Status:** Draft | **Amends:** TRD v1.3 | **Implements:** www-landing PRD v0.1 §7

This addendum records the TRD changes required by the www-landing PRD. It amends TRD v1.3 in place; the changes below fold into the next full TRD revision (v1.4).

---

## §3 Repository Structure — addition

```
├── apps/
│   ├── www/                       # SvelteKit — root-domain landing (Phase 0 launch unit)
│   │   └── src/
```

`apps/www` is the platform's public face on the apex domain. It is part of the Phase 0 launch unit (joint partai + mufakat) but is an independent static deployment — not technically gated on either app.

## §4 Distribution & Deployment — addition

```
www:
  Build command: pnpm build
  Output directory: .svelte-kit/cloudflare
  Adapter: @sveltejs/adapter-cloudflare
  Custom domain: alternatif.space   (apex)
```

- Fully prerendered: `prerender = true` on the root layout. Only the Clerk widget hydrates client-side (PRD §7 — "static, no JS beyond Clerk + minimal interaction").
- Environment variables: `PUBLIC_CLERK_PUBLISHABLE_KEY` only. **No Supabase dependency at launch** — a live stat ("X partai aktif") is deferred because it would break full prerendering.
- **Native shell:** the landing is *not* added as a Capacitor tab — app users are past the marketing layer.

## §5 Auth & Cross-Subdomain Session — addition

The landing is an **auth entry point**:

- Same Clerk instance and env var set as the other apps. The apex sits *inside* the `.alternatif.space` cookie scope, so session detection works without satellite-domain configuration.
- **Client-only Clerk** — unlike partai/mufakat there is no `withClerkHandler` server hook (the app is fully prerendered; server hooks never run at request time). `<ClerkProvider>` initialises clerk-js after hydration, which reads the shared session cookie. Embedded components use `routing="hash"` because no catch-all route exists for path routing.
- Signed out → embedded `<SignUp />` in the hero with `forceRedirectUrl` → `https://partai.alternatif.space/onboarding` (partai owns post-signup onboarding; the landing never duplicates it). Secondary `/masuk` route hosts `<SignIn />` with `forceRedirectUrl` → partai root.
- Signed in → all auth CTAs swap to **"Lanjut ke partai →"** (partai's root decides between party page and onboarding; www cannot do a membership lookup — no Supabase).

## Out of scope (unchanged from PRD §8)

Blog/press kit, English version, live platform statistics, newsletter capture, analytics tooling (checklist §10).
